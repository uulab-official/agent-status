# Data model

Source of truth: [`crates/core/src/types.rs`](../crates/core/src/types.rs).
This document explains the *why* behind the shapes; read the file itself for
the exact fields.

## `ProviderStatus` ‚Äî the only thing downstream consumers see

```rust
pub struct ProviderStatus {
    pub provider_id: String,
    pub display_name: String,
    pub state: ConnectionState,
    pub limits: Vec<LimitWindow>,     // zero or more ‚Äî Claude reports 2 at once: session + weekly
    pub models: Vec<ModelInfo>,
    pub cost: Option<CostSnapshot>,
    pub observed_at: String,          // RFC 3339 ‚Äî when this reading was taken
    pub detail: Option<String>,       // free-form debug string, e.g. "fetch_status() not implemented"
}
```

The popover and notification engine both consume `ProviderStatus`
exclusively (see `view_model.rs::build_popover_view_model` and
`NotificationEngine::evaluate`). Neither special-cases a provider by id ‚Äî if
the popover view model ever needed an `if status.provider_id == "claude"`,
that would be a sign something that belongs in the Claude plugin leaked into
shared code.

## `LimitWindow` ‚Äî one quota, one period

A provider can report several of these at once. Claude's 5-hour rolling
window and its weekly window would be two separate `LimitWindow` entries,
not one window with two numbers bolted on ‚Äî that's what lets the popover
render "Resets in 2h 14m" and "41% this week" as independent bars.

Fields worth calling out:
- `limit: Option<f64>` ‚Äî `None` means "no fixed ceiling" (OpenRouter's
  pay-as-you-go keys don't have a hard cap to bar-chart against ‚Äî see
  `provider-openrouter`'s handling of a `null` limit).
- `resets_at: Option<String>` ‚Äî optional because not every window resets on
  a timer (a one-time credit grant doesn't).
- `confidence: Confidence` ‚Äî see [confidence.md](confidence.md). Required,
  not optional ‚Äî every window must say how sure we are about it.

## `ConnectionState` ‚Äî eight states, not a boolean

The temptation is `online | offline`. Eight states exist because "the
provider is reachable but you should still look at it" is a real, common
state that a boolean can't express:

| State | Meaning |
|---|---|
| `Online` | Normal, plenty of headroom |
| `Busy` | Reachable but degraded (queueing, elevated latency; also how Ollama reports "a model is currently loaded") |
| `RateLimited` | Hard limit hit ‚Äî requests are being rejected right now (how OpenRouter reports usage ‚â• limit) |
| `ResetSoon` | Comfortably under budget, but a reset is imminent |
| `Offline` | Not reachable / not configured / disabled |
| `Waiting` | Actively fetching a fresh reading |
| `Updating` | Plugin is mid self-update (token refresh, model list refresh) |
| `Unknown` | Fetch failed, or `fetch_status()` isn't implemented yet ‚Äî see `detail` |

## `CostSnapshot` vs. `LimitWindow`

Cost and usage are modeled separately because they don't share a reset
cadence or a unit. A `LimitWindow` with `unit: "usd"` and a fixed period
still makes sense (a monthly spend cap), but *unbounded* spend tracking
(OpenRouter's running balance) doesn't fit "used out of a limit" ‚Äî it's just
a number that goes up. `CostSnapshot` exists for that: `today` / `this_week`
/ `this_month` / `credits_remaining`, no `limit` field at all.

## `AgentNotification`

Generated by `agent-notifications`' `NotificationEngine`, not by plugins
directly (a plugin *can* queue its own via `BasePluginState::queue_notification`
for provider-specific events ‚Äî the trait method is
`ProviderPlugin::drain_notifications`, with a default empty implementation ‚Äî
but threshold-crossing notifications like "10% left" or "resets in 15
minutes" are computed centrally so every provider gets the same thresholds
without reimplementing the logic). The `reason` field is a stable,
machine-parseable key (`"claude:session:low_10"`) used for dedup ‚Äî see
[architecture.md](architecture.md#data-flow-at-runtime).

## Where this is persisted

`crates/database/src/schema.sql` mirrors this model into SQLite:
`usage_history` samples `LimitWindow` readings over time (this is what will
power the Timeline view on the roadmap ‚Äî nothing writes to this table yet),
`cost_history` samples `CostSnapshot`, `notifications` stores
`AgentNotification` with a `dismissed_at` for read-state, and `settings` is
a generic key-JSON store used today for `trayMode` and `launchAtLogin` (see
`agent-database`'s `get_setting`/`set_setting`). See the schema file itself
for the full DDL ‚Äî it's short and meant to be read directly rather than
summarized further here.
